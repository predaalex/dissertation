\chapter{Facial Emotion Recognition Model}

\section{Dataset Description: FER2013 Enhanced}

The facial emotion recognition model was trained using FER2013 Enhanced, available on Hugging Face as \textit{abhilash88/fer2013-enhanced} \cite{fer2013_enhanced_2025}. This dataset is based on the original FER2013 dataset, which was introduced in the Challenges in Representation Learning competition \cite{goodfellow2015challenges}. Instead of using the original FER2013 files directly, my project uses the enhanced version because it already provides dataset splits and extra information that is useful during training.

FER2013 Enhanced contains 35,887 grayscale face images. Each image has a resolution of 48x48 pixels and is labeled with one of seven emotion classes: angry, disgust, fear, happy, sad, surprise, and neutral. The dataset is already split into training, validation, and test sets. The split sizes are shown in Table~\ref{tab:fer2013-enhanced-splits}.

\begin{table}[h]
    \centering
    \begin{tabular}{lrr}
        \hline
        Split & Number of images & Percentage \\
        \hline
        Train & 25,117 & 70.0\% \\
        Validation & 5,380 & 15.0\% \\
        Test & 5,390 & 15.0\% \\
        \hline
        Total & 35,887 & 100.0\% \\
        \hline
    \end{tabular}
    \caption{FER2013 Enhanced dataset split sizes.}
    \label{tab:fer2013-enhanced-splits}
\end{table}

The dataset is not balanced across classes, as shown in Figure~\ref{fig:fer2013-enhanced-overview}. The happy class has the largest number of samples, while disgust has far fewer samples than the other classes. This matters because a model trained directly on an imbalanced dataset can learn to prefer the larger classes. For this reason, class imbalance is considered during the experimental process.

FER2013 Enhanced also provides a quality score for each image. This score is computed from four image properties: edge content, contrast, focus or sharpness, and brightness balance. Edge content and contrast each contribute 30\% of the score, focus contributes 25\%, and brightness balance contributes 15\%. The quality score distribution is also shown in Figure~\ref{fig:fer2013-enhanced-overview}.

The quality score is useful because it allows experiments that compare training on all samples with training only on samples above a minimum quality threshold. The experiments use two filtering thresholds, 0.35 and 0.5. The threshold 0.35 is a permissive filter that removes the weakest images while keeping most of the dataset. The threshold 0.5 is stricter and tests whether training on cleaner images can improve the model, even if fewer samples remain. These thresholds are not treated as fixed rules for the dataset. They are used to measure the trade-off between data quantity and data quality. Table~\ref{tab:quality-threshold-retention} shows how many samples remain after each threshold.

\end{table}


\section{Data Loading, Preprocessing, and Augmentation}

The dataset is loaded in the training script using the Hugging Face \textit{datasets} library. The default dataset name is \textit{abhilash88/fer2013-enhanced}, and the files are cached locally. The same loading function is used for the training, validation, and test splits.

Before an image is passed to the model, it is converted to RGB. The dataset images are grayscale, while the pretrained MobileNetV2 backbone expects three input channels. The label is read from the \textit{emotion} field and used as the target class.

The training script can also limit the samples used in an experiment. A dataset fraction can be selected for shorter runs, and a minimum quality score can be applied to remove low-quality samples. These options make it possible to compare different training settings without modifying the dataset files.

Different image transformations are used for training and evaluation, as shown in Table~\ref{tab:image-transforms}. The training split uses random crop, horizontal flip, and rotation. The validation and test splits use deterministic transformations, so reported metrics are not affected by random input changes.

\begin{table}[h]
    \centering
    \begin{tabular}{llll}
        \hline
        Step & Training & Validation \& Test \\
        \hline
        Resize & 232x232 & 232x232 \\
        Crop & Random 224x224 & Center 224x224 \\
        Horizontal flip & Yes & No \\
        Rotation & Up to 10 degrees & No \\
        Tensor conversion & Yes & Yes \\
        Normalization & ImageNet stats & ImageNet stats \\
        \hline
    \end{tabular}
    \caption{Image transformations used during training and evaluation.}
    \label{tab:image-transforms}
\end{table}

The random transformations are applied only during training. Validation and test images are processed with the same fixed resize, crop, tensor conversion, and normalization steps in every run.

\section{Model Architecture}

The classifier uses MobileNetV2 as the backbone. MobileNetV2 was designed as a lightweight convolutional neural network for mobile and resource-limited environments. Its main contribution is the use of inverted residual blocks with linear bottlenecks, combined with depthwise separable convolutions \cite{sandler2018mobilenetv2}. This design reduces the number of operations while keeping good classification performance.

\begin{figure}[h]
    \centering
    \includegraphics[width=\linewidth]{MobileNetV2_architecture_overview.png}
    \caption{MobileNetV2 architecture overview.}
    \label{fig:mobilenetv2-architecture}
\end{figure}

MobileNetV2 was selected because the project is intended to run locally and was trained on a single GPU. The model is also used for live webcam classification, so inference speed is important. The assistant has other local components, such as speech-to-text and LLM inference, so the emotion model should not be unnecessarily large. TorchVision reports MobileNetV2 with 3.5M parameters and 0.30 GFLOPS for ImageNet inference \cite{torchvision_models}. Table~\ref{tab:backbone-comparison} shows several possible backbones with similar or commonly used sizes.

The comparison shows that MobileNetV2 is small, has low computational cost, and still offers reasonable pretrained classification performance. Some alternatives have better ImageNet accuracy, but they also increase parameter count or computation. For my project, MobileNetV2 gives a suitable balance between model size, inference speed, and transfer learning performance.

\section{Training Pipeline}

The training pipeline was implemented in PyTorch and all experiments were run on a local machine with an NVIDIA GeForce RTX 4070 Ti GPU with 12 GB of VRAM. This hardware was sufficient for training MobileNetV2 and matched the local execution direction of the application.

Training starts from the ImageNet-pretrained MobileNetV2 weights. The original classifier is replaced with a dropout layer followed by a linear layer with seven outputs, one for each emotion class. The rest of the network is reused as a visual feature extractor.

The pipeline supports two training strategies. In the baseline strategy, the MobileNetV2 feature extractor is frozen and only the new classification head is trained. In the fine-tuning strategy, the model is trained in two stages. First, only the classification head is trained. Then, the last MobileNetV2 blocks are unfrozen and training continues with a smaller learning rate.

The optimizer used in both strategies is AdamW. The loss function is cross-entropy, with optional class weights when class-weighted loss is enabled. These weights are computed from the class distribution of the training split. The scheduler monitors validation macro-F1 and reduces the learning rate when the score stops improving.

After each epoch, the model is evaluated on the validation split. The training loop records loss and accuracy, while validation also computes macro-F1 and per-class precision, recall, and F1-score. The best checkpoint is saved whenever validation macro-F1 improves. Early stopping is used when validation macro-F1 does not improve for several consecutive epochs.

Automatic mixed precision is enabled to reduce memory usage. Gradient clipping is also applied before the optimizer step to keep training stable.
